# Switch job_listings primary key to composite (source_id, id)

## Context

The most recent commit on `main` (`92bfdf6 Move Greenhouse to Backend Cron + Queue`) introduced an asymmetric ID convention in `job_listings`:

- Greenhouse rows: `id = "greenhouse_{raw_id}"` (prefix added in `src/backend/api/services/greenhouse_client.py:102`).
- Google / Apple / Microsoft rows: raw upstream id, no prefix.

The prefix exists because `job_listings` has a single-column PK on `id`, so cross-source uniqueness is enforced by convention (each source picks a non-colliding id format). This is a "stringly-typed compound key" — the `id` column carries two pieces of information (identity + source) glued with an underscore.

The table already has dedicated `company` (Text NOT NULL) and `source_id` (Text NOT NULL) columns sitting next to `id` (`src/backend/api/db_models.py:38-63`). The correct design is to lift uniqueness into the schema: `PRIMARY KEY (source_id, id)`. Each source then stores its raw upstream id, and the database enforces the actual domain rule ("ids are unique per source"). Side benefits: no per-source "do we prefix?" decision (today only Greenhouse prefixes), and a future ATS source whose id space overlaps with an existing source's id space is handled for free.

`source_id` (not `company`) is the right second column because it matches the actual uniqueness boundary the upstream system guarantees. Greenhouse guarantees uniqueness across all 45 boards (verified by the pre-flight collision check in migration `e6cbbb3c2f17`), so the namespace is "all of Greenhouse" — not "Greenhouse + specific company". `(company, id)` would be over-specified and would silently allow two different scrape sources to both claim `(company='anthropic', id='12345')`.

## Decisions Locked

| Decision | Choice |
|---|---|
| PK shape | Composite `(source_id, id)` on `job_listings`. |
| Greenhouse id format going forward | Raw upstream numeric as string. No `greenhouse_` prefix. |
| Other sources (Google / Apple / Microsoft) | No change — they already emit raw upstream ids. |
| Second PK column | `source_id` (matches upstream uniqueness boundary). Not `company`. |
| Migration mechanism | **`alembic revision --autogenerate`** from the repo root, then hand-edit `upgrade()`/`downgrade()` bodies **only** to add pre-flight guards and the data UPDATE. **Never hand-write the migration file from scratch.** Modify the SQLAlchemy model first, let Alembic detect the schema diff, then layer the data-rewrite ops into the autogenerated skeleton. |
| Migration pre-flight guard | Mirror `e6cbbb3c2f17` — `DO $$ … RAISE EXCEPTION` block counting collisions before any destructive write. Symmetric guard on the downgrade side. |
| Bulk `WHERE id IN (…)` scoping | Add `source_id = %s AND` for symmetry; callers already operate on rows from one source per run. |
| HTTP route shape | `GET /api/jobs/{source_id}/{job_id}`. Old `/{job_id}` shape is removed (not preserved via id parsing). |
| Frontend / Vercel proxy impact | None. `api/jobs.ts` proxies list queries only; `backendScraperClient.ts` does not use the by-id route. |
| Deploy ordering | Standard Railway lifespan-hook deploy. Migration runs before worker starts (already wired in `src/backend/api/main.py`). No special runbook beyond DEPLOY.md. |

## Repo Constraints (must follow)

- **Alembic autogenerate only — non-negotiable.** The workflow is:
  1. Modify `src/backend/api/db_models.py` (drop `primary_key=True` from `id`, add `PrimaryKeyConstraint("source_id", "id")` to `__table_args__`).
  2. Run `alembic revision --autogenerate -m "job_listings composite (source_id, id) primary key"` from the directory that resolves `alembic.ini` (per `src/backend/api/tests/test_migration_companies.py:26-28`, that is the repo root).
  3. Hand-edit **only the bodies of** `upgrade()` and `downgrade()` in the autogenerated file to insert the pre-flight `DO $$ … RAISE EXCEPTION` guards and the `UPDATE job_listings SET id = regexp_replace(…)` data rewrite around the auto-detected `ALTER TABLE` ops. Combine the DROP CONSTRAINT and ADD PRIMARY KEY into one `ALTER TABLE … DROP CONSTRAINT …, ADD PRIMARY KEY …` statement per the combined-ALTER rule.
  4. **Never** create a migration file by hand from a template. **Never** add files under `scripts/shared/migrations/` (that directory is frozen — Alembic is the only migration runner).
- **Bare table names.** Post-`115476c` the table is `job_listings` in every environment; test isolation is via `PYTEST_SCHEMA` + `SET search_path`. Do not introduce `_{env}` suffixes.
- **No full-table rewrites.** A single-column-to-composite PK swap is metadata-only when no `USING` clause is needed. `ALTER TABLE job_listings DROP CONSTRAINT job_listings_pkey, ADD PRIMARY KEY (source_id, id)` qualifies — it rewrites the PK index only, not the heap. The data UPDATE that strips `greenhouse_` is row-targeted (`WHERE source_id = 'greenhouse_api' AND id LIKE 'greenhouse_%'`) and touches only that subset of rows.
- **Combined ALTER TABLE rule.** Per `docs/implementations/alembicMigration/DEPLOY.md`, the DROP CONSTRAINT and ADD PRIMARY KEY MUST be one statement (single `ALTER TABLE … DROP CONSTRAINT …, ADD PRIMARY KEY …`) so Postgres rebuilds the PK index once.
- **Each unit independently committable.** All tests (`cd src/backend && pytest`, `npm run type-check`, `npm test` from repo root) must pass at HEAD after every unit's commit, not just at the end of the PR.

## Architecture

A small, self-contained schema migration plus the call-site updates that ride with it. The migration runs in the existing FastAPI lifespan hook (`apply_alembic_migrations(database_url)` → Procrastinate `open_async` → worker start) on Railway deploy. No new infrastructure, no new services, no new env vars. Once the PK swap lands, every `ON CONFLICT (id)` / `WHERE id = %s` / `WHERE id IN (…)` site in the codebase must reference `(source_id, id)` in lock-step, which is why the schema change, the query layer, the Greenhouse transformer, and the `get_job_by_id` service helper all land in one atomic unit (Unit 1). Unit 2 then narrowly cuts over the by-id HTTP route to the new path shape. Unit 3 documents the deploy.

## Shared Contracts

These are the frozen interfaces every unit must respect. Once Unit 1 lands they cannot drift.

**1. `job_listings` schema (after Unit 1):**

- `PRIMARY KEY (source_id, id)` — composite. Old single-column PK on `id` is dropped.
- Greenhouse rows: `source_id = 'greenhouse_api'`, `id` = raw upstream numeric id as a string (e.g. `'12345'`). No `greenhouse_` prefix.
- Google / Apple / Microsoft rows: unchanged. `id` is the raw upstream id; `source_id` is whatever marker those scrapers already emit today.

**2. HTTP routes (after Unit 2):**

- `GET /api/jobs/{source_id}/{job_id}` — by-id lookup. Returns 404 if no row matches the composite key. Old `GET /api/jobs/{job_id}` is removed.
- `GET /api/jobs?company=…&status=…&limit=…&offset=…` — list endpoint, unchanged.

**3. Backend service helper (after Unit 2):**

```python
def get_job_by_id(conn: Connection, source_id: str, job_id: str) -> dict | None
```
in `src/backend/api/services/database.py`.

**4. `scripts/shared/database.py` query layer (after Unit 1):**

- `upsert_jobs_batch`, `upsert_job`, `insert_jobs_batch` — all `ON CONFLICT` clauses reference `(source_id, id)`.
- `get_job_by_id(conn, source_id, job_id)` — composite lookup.
- `update_last_seen`, `increment_consecutive_misses`, `mark_jobs_closed`, `get_jobs_exceeding_miss_threshold` — all bulk `WHERE id IN (…)` clauses additionally take a `source_id` arg and use `WHERE source_id = %s AND id IN (…)`.
- `reactivate_job(conn, source_id, job_id, timestamp)` — composite single-row update.

## 1. Test harness conventions (shared across all 8 files)

### Real store over dispatch spies (Recommendation A)
For every test that asserts a Redux dispatch, construct a real store via `createTestStore` / `renderWithProviders`, perform the interaction, then read `store.getState()` and assert the slice state is correct. This is stronger than a dispatch spy — it verifies the reducer ran end-to-end with the right payload, not just that `dispatch` was called with an action whose type string matches.

Pattern:
```ts
const { store } = renderWithProviders(<Component />, { preloadedState });
await user.click(/* ... */);
expect(store.getState().graphFilters.filters.timeWindow).toBe('7d');
```

### When a spy is unavoidable (Fallback B)
For the `SyncFiltersButton` component in isolation and for the leaf shared controls (`TimeWindowSelect`, `SoftwareOnlyToggle`, `MultiSelectAutocomplete`, `SearchTagsInput`) — which take `onChange`/`onAdd`/`onRemove`/`onToggleMode`/`onClick` callbacks as **props** — use `vi.fn()` and assert call args directly. No store needed. Render with `render` from `@testing-library/react` (no providers needed) for speed.

### "Does not re-render unrelated siblings" — pragmatic reinterpretation
PLAN.md §Unit 9 says the bar should not re-render unrelated siblings. Do **not** reach for react render counters (brittle, internal-implementation coupled). Instead assert:
- After dispatching an action that touches one filter slice, the **other** filter slice's state is byte-identical (`expect(store.getState().listFilters).toBe(snapshotListFiltersBefore)` for graph-vs-list independence).
- Within one bar, after a single control change, only the intended sub-field of that slice's `filters` object is different from the initial state; all others (`searchTags`, `softwareOnly`, etc.) are equal to the initial values.

This is a stronger guarantee than render-count equality, and it's what the "filter independence" PLAN contract actually wants.

### MUI Autocomplete interaction reference
`MultiSelectAutocomplete` and `SearchTagsInput` both wrap MUI `Autocomplete`. For MUI Autocomplete tests:
- To open: `await user.click(screen.getByRole('combobox'))` or focus + keyboard `{arrowDown}`.
- Options appear under `role="listbox"` with each item `role="option"`.
- To type (freeSolo): `await user.type(input, 'senior{enter}')`.
- To remove chip: find the chip's delete icon (it's an `svg` sibling with `data-testid="CancelIcon"`); click its parent button.

### No `fireEvent.change` on MUI Select
MUI `Select` does not respond to `fireEvent.change` on the hidden input. Use `userEvent.click` to open + `userEvent.click` on the option, OR use `fireEvent.mouseDown` on the select button.

## 6. Risks / HEAD deviations (inline; DO NOT edit PLAN.md)

1. **PLAN.md "does not re-render unrelated siblings" bullet** is interpreted as "unrelated slice state reference-preserved after a slice-local action" (§1 conventions). Reason: render-counter assertions on MUI subtrees are brittle and don't actually prove independence; reducer-identity is the real contract.

2. **`syncGraphToList` / `syncListToGraph` are thunks**, not plain actions (confirmed in `features/filters/syncActions.ts`). Testing these requires a real store because the thunk calls `getState()` at dispatch time. A dispatch spy would miss the cross-slice copy.

3. **`MultiSelectAutocomplete` is conditionally rendered** inside `GraphFilters`/`ListFilters` (`availableLocations.length > 0` / `availableDepartments.length > 0`). Bar tests must seed jobs with `location` and `department` fields or those tests silently skip the branch. This is called out in the bar-test specs above.

4. **`RecentJobsFilters` uses a different query endpoint** (`getAllJobs`) than the Companies-page bars (`getJobsForCompany`). The seed helpers differ accordingly.

5. **Default `recentJobsFilters.filters.timeWindow` is `'3h'`** — fresh `Date.now()` `createdAt` values will fall inside this window, but if the seeded data ages (e.g., via test snapshotting), the "available companies/locations" may go empty. To eliminate flakiness, preload `recentJobsFilters.filters.timeWindow = '30d'` in RecentJobsFilters tests.

6. **`enabledCompanies` slice state** is required by `selectEnabledByCompanyId` inside `RecentJobsFilters`. Default `createTestStore` uses the slice's real initial state (`{ ids: null, loading: false, error: null, activeLoadRequestId: null }`), which means "all enabled" semantics hold. If tests behave oddly, preload explicitly.

7. **No source files are modified** in this unit. If coverage on one of the 8 source files is unreachable >=85% for a real reason (e.g., a prop default that is never exercisable), document it in the commit body and accept the lower bar for that one file. Do **not** refactor the source to make it testable in Unit 9.

8. **MUI keyboard interaction for chip removal**: `userEvent.keyboard('{Backspace}')` on a focused Autocomplete input removes the last chip. If the test flakes, click the input's text field directly (not the wrapper) before the keypress.

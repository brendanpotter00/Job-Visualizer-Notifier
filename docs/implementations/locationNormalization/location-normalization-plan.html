Give every job a canonical location

ATS APIs hand us strings like "sf", "San Francisco, CA", "SF Bay Area", "San Francisco, CA / New York, NY / Remote". Today job_listings.location TEXT stores these raw — no canonical form, no geo, no clean way to answer "show me all SF-or-remote jobs." This adds a canonical model populated by an async normalization pipeline that rides the existing Procrastinate worker.

🎯 Goal

A canonical location model + an async pipeline so filtering, grouping, and (later) geo all key off normalized data, not raw text.

📦 Current state

One free-text location column. No dedupe, no multi-location, no geo. Filters resort to brittle LIKE.

✅ Desired

Every job_listings row → 0+ canonical location rows via a job_locations join table. Multi-location is first-class.

Approach — a two-tier cascade on the existing worker

Tier 1

Postgres alias cache

location_aliases keyed on the pre-normalized raw string. After warmup, ~90%+ are cache hits — zero LLM calls.

Tier 2

Claude Haiku 4.5

Only on cache misses. Returns structured JSON for one or more canonical locations; result is cached so the next occurrence is free.

Decoupled: normalization is a new normalize queue on the worker that already exists — its own retry/concurrency budget, so a slow LLM never blocks ATS fetches.

Schema — your 5 questions answered inline

📌 "What if a job has multiple locations?" — it is NOT one field on the job row

You're right that stuffing a list into the job record would be wrong — and we don't. The job_listings row is never given a normalized-location field. The only column we add to it is normalization_status. The locations live in a separate join table, job_locations — one row per location. A job with two locations produces two job_locations rows, each a real foreign key to a canonical locations row.

job_listings 1 row  ──<  job_locations N rows (the join)  >──  locations canonical, shared

Concretely, the multi-location job from Example B, across all three tables:

locations — the canonical rows those FKs point at (shared by other jobs)

So "how many locations does a job have?" is just SELECT count(*) FROM job_locations WHERE job_listing_id = …, and "all SF-or-remote jobs" is an indexed join — never a LIKE over a packed string. This is standard third-normal-form: the job row holds no list, the join table holds the relationships. 

class Location(Base):   # "locations"
  id              PK
  canonical_name  Text NOT NULL  # display string
  kind            Text NOT NULL  # city|region|country|remote
  city/region/country  Text NULL  # structured parts
  remote_scope    Text NULL      # NULL|global|us|eu|...
  lat / lng       Float NULL     # geo (see note)
  UNIQUE(kind, city, region, country, remote_scope)

❓ kind vs city/region/country?

kind is the discriminator — it tells you which structured fields are meaningful. The others are the actual parts.

kindcityregioncountryremote_scope
citySan FranciscoCAUS—
region—CAUS—
country——US—
remote———us

❓ Why is remote_scope in the UNIQUE key?

Because Remote (US), Remote (EU), and Remote (global) are distinct canonical locations — they filter differently. Leaving remote_scope out of the key would collapse them into one row, so a "remote US-only" job and a "remote anywhere" job would be indistinguishable. It's part of canonical identity. (For non-remote kinds it's NULL, and Postgres treats NULLs as distinct — fine here because kind+parts already disambiguate those.)

❓ Where do lat/lng come from? Honestly: nowhere reliable in v1. Haiku can emit approximate city centroids, but coordinates are the thing LLMs hallucinate most. Recommendation: keep the columns (nullable) but drop lat/lng from the v1 LLM ask and leave them NULL; add a dedicated geocoding pass (Nominatim/Mapbox over canonical_name) as the deferred "Geo features" follow-up. → decision Q3
❓ How is locations initially instantiated? It isn't seeded — it starts empty and is populated lazily: every Tier-2 cache miss UPSERTs the canonical rows it produces (ON CONFLICT DO NOTHING dedupes). The cache and canonical set co-grow with real traffic. Optional: seed a handful of obvious ones (SF, NYC, Remote-US) to skip first-call latency — not required. 
class LocationAlias(Base):  # "location_aliases"
  raw_text   PK   # pre-normalized cache key
  source / confidence
 
class AliasLocation(Base):  # "alias_locations"
  raw_text FK + normalized_location_id FK   (PK)
  position INT   # order within the raw string
 
class JobLocation(Base):    # "job_locations"
  job_listing_id  FK → job_listings.id   (PK)
  normalized_location_id  FK → locations.id (PK)
  is_primary  BOOL

❓ Is JobLocation a new table? Isn't a comma-separated text field bad?

Yes, new table — and no comma-separated text anywhere. The mockup above compressed it; to be unambiguous:

  • job_locations is a proper many-to-many join table: one row per (job, location) pair.
  • A job with 3 locations → 3 rows, each a real FK to a locations row. Indexed, joinable, no string parsing at query time.
  • "show me all SF-or-remote jobs" becomes a clean indexed join, not a LIKE.
  • is_primary marks the first/primary location for default display.

This is exactly the efficient, normalized shape — the raw comma-string never makes it past Tier 2.

Additive column: JobListing.normalization_status Text NULL — NULL | pending | done | failed. IS NULL = "never attempted", the signal the safety-net cron queries.

Tier 2 — the Haiku prompt your Q

Lives in services/llm_client.py. Short system prompt + structured output (tool/JSON mode), 2–3 few-shot examples. Draft:

You normalize raw job-location strings into canonical locations.
Return a JSON array; one object per distinct location in the input.
Fields: canonical_name, kind(city|region|country|remote),
        city, region, country, remote_scope, confidence(0..1).
Rules: expand abbreviations ("sf"→San Francisco, CA, US).
Split multi-location strings ("A / B / Remote") into N objects.
"Remote (US)"→kind=remote, remote_scope=us. Do NOT emit lat/lng.
 
"San Francisco, CA / Remote (US)"
[{canonical_name:"San Francisco, CA, US", kind:"city", city:"San Francisco", region:"CA", country:"US", confidence:0.98},
 {canonical_name:"Remote (US)", kind:"remote", remote_scope:"us", confidence:0.95}]

Model id claude-haiku-4-5-20251001, 10s timeout, retries via Procrastinate's RetryStrategy (not tenacity). Adopted: the prompt returns a per-location confidence, and a value below a threshold marks status='failed' for review instead of caching a guess (the confidence-floor mitigation in F2 cache poisoning).

F1

Connection-hold & worker-concurrency under the 10s LLM call root cause + fix you asked for

What caused the prior incident (docs/incidents/2026-05-17-recent-jobs-pool-exhaustion.md): the Greenhouse→backend migration (#110) grew the Recent Jobs page from 3 to 49 simultaneous /api/jobs?company=X reads, each holding one of the 15 pool slots for its full duration. The semaphore's 5s acquire timeout (dependencies.py:76) elapsed on ~20 of them every load. The headline lesson: the failure mode is connections held across slow work, and the fix was structural (batch the fanout), not growing the pool — Railway's 3 GB ceiling + Playwright memory pressure made "raise DB_POOL_MAX to 50" explicitly the wrong answer.

Correction to rev 1: I wrongly said normalize "rides the same 15-slot pool." It doesn't — like every existing fetch task, it opens its own standalone psycopg2 connection (the task files note "separate from the FastAPI request pool… won't show up in pool metrics"). So it can't time out /api/jobs reads. The only residual risk is total concurrent connections vs Railway memory — and the design below neutralizes even that without capping throughput, so normalization still scales with the whole worker pool.

Suggested design revised per your pushback — scale-friendly

You're right: a hard concurrency=2 cap is anti-scale and not worker-agnostic. Dropped. The load-bearing fix is the connection discipline, which is exactly what makes high concurrency safe:

  1. Never hold a DB connection across the Haiku call. Read + Tier 1 on a short-lived connection → release it → 10s LLM call with no connection open → fresh short connection to write. The key consequence: a normalize task holds a connection only for two brief writes, not for its ~10s lifetime — so 500 in-flight normalize tasks ≠ 500 open connections. The exact thing the incident punished (connections held across slow work) structurally can't happen.
  2. Scale with the pool — no cap. normalize is just another queue the existing worker drains, at whatever concurrency / however many worker replicas you run. Normalization tasks are independent (one job_id each, order doesn't matter), so they parallelize freely and a big backlog drains as fast as the pool allows. Procrastinate is horizontally scalable — 1,000 workers would use all 1,000.
  3. If you ever need to shield ATS fetches from a huge normalize backlog, the worker-agnostic way is a separate worker replica/process dedicated to the normalize queue (scales independently) — never a coroutine cap. Not needed for v1, because (1) makes each task cheap.
  4. Only real ceiling: total concurrent DB connections vs Railway memory. Thanks to (1) that tracks active write-moments, not in-flight task count, so it stays low even under heavy parallelism. Don't preemptively grow DB_POOL_MAX — the incident showed that's the wrong lever.
F2

Cache poisoning → a correction workflow (your audit-agent idea) approved

A wrong LLM result is cached and reused (no per-alias invalidation today). Three mitigations, layered:

  • Confidence floor adopted: the LLM call + prompt return a per-location confidence; below a threshold, set status='failed' instead of caching a guess — surfaces it for review rather than silently poisoning the cache.
  • Manual override persists: a correction writes a location_aliases row with source='manual' that wins over the llm one (the source column already exists for exactly this). Add a small admin endpoint for it (→ Unit 8).
  • Claude location-audit skill (your suggestion): a scheduled/triggerable agent that queries the prod DB for suspect canonical rows (e.g. kind='country' where a city was expected, low-confidence aliases, orphans) and corrects each via the admin endpoints. Document this in the ops/DEPLOY docs (you flagged this). Tracked as a follow-up — see Out of Scope.
F3 · Bulk re-normalize is break-glass. Per your call: keep re-normalize-all but treat it as a rarely-used, gated, throttled/batched escape hatch. Day-to-day correction is the targeted audit-agent path (F2) — no thundering herd of Haiku calls.
